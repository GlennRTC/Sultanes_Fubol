**PART 2 — RLS Security Audit (10 minutes)**

Open the Supabase Dashboard SQL Editor. You will run queries in two contexts:
"impersonate user" mode for a regular user, and "impersonate user" mode for the admin.

To impersonate in SQL Editor: use the "Run as" dropdown in Supabase Dashboard.
Alternatively, open the browser console while logged in as the target user and run:

    supabase.auth.getUser().then(r => console.log('uid:', r.data.user?.id))

Then use these test queries from RESEARCH.md Q8:

**As a regular (non-admin) user:**

    -- Should return 1 (own row only):
    supabase.from('profiles').select('id').then(r => console.log('profiles count:', r.data?.length))

    -- Should return 0 (own predictions only, no other users'):
    supabase.from('predictions').select('*').then(r => {
      const crossUser = r.data?.filter(p => p.user_id !== YOUR_USER_ID);
      console.log('cross-user predictions:', crossUser?.length); // should be 0
    })

    -- admin_logs: should return 0 rows (admin-only):
    supabase.from('admin_logs').select('*').then(r => console.log('admin_logs count:', r.data?.length))

    -- token_transactions: should return 0 rows (admin-only):
    supabase.from('token_transactions').select('*').then(r => console.log('tx count:', r.data?.length))

    -- Admin function call: should fail with 'not_admin':
    supabase.rpc('admin_block_user', { p_target_user_id: 'any-uuid', p_blocked: true })
      .then(r => console.log('error:', r.error?.message)) // should be 'not_admin'

**Expected results for regular user:**
- profiles count: 1 (own row)
- cross-user predictions: 0
- admin_logs count: 0
- token_transactions count: 0
- admin_block_user error: 'not_admin'

**As admin user:**
- profiles count: N (all users)
- admin_logs count: N (all log entries)
- token_transactions count: N (all transactions)

Mark each result: PASS or FAIL. If any result is FAIL, stop and describe the failure — do not proceed to the smoke test until RLS is corrected.

**PART 3 — E2E Smoke Test (15 minutes)**

Follow this sequence end-to-end. You need two browser windows: admin (Window A) and regular user (Window B).

Step 1 — Admin creates a pool:
  Window A: Log in as admin → navigate to /admin/apuestas
  Fill in: Question = "¿Quién ganará el grupo A?", Type = "Opción múltiple",
    Deadline = any future date, Options = ["México", "Estados Unidos", "Canadá"]
  Click "Crear pool". Confirm success message appears with a pool ID.

Step 2 — Regular user bets:
  Window B: Log in as a regular user → navigate to /apuestas
  Confirm the new pool appears in "Activas" section.
  Click the pool card → BetModal opens.
  Select "México" and enter 50 fichas.
  Click "Confirmar apuesta" → confirm "Confirmar".
  Confirm: Navbar token balance decreased by 50. Pool card shows bet badge.

Step 3 — Admin resolves the pool:
  Window A: Navigate to /admin/apuestas → find the pool in "Pools Activos".
  Select "México" as winning option. Click "Resolver pool".
  Confirm success message.

Step 4 — User sees payout:
  Window B: Reload /apuestas.
  Confirm: The pool is now in "Cerradas" section with "Resuelta" badge.
  Confirm: The pool card shows "Ganaste: +50 fichas" badge (bet on winning side).
  Confirm: Navbar token balance is back to (starting balance - 50 + payout).
    Because user was the only bettor, payout = 50 (all the pool tokens go back to them).

Step 5 — admin_logs verification:
  Window A (admin): Open browser console, run:
    supabase.from('admin_logs').select('*').order('created_at', { ascending: false }).limit(5)
      .then(r => console.log(r.data))
  Confirm: At least two entries exist: one 'pool_created' and one 'pool_resolved'.
  Confirm: 'pool_bet' is NOT in admin_logs (it's in token_transactions, not admin_logs).

Step 6 — Password reset test (optional but recommended):
  Window A: Navigate to /admin/usuarios.
  Find the regular user's row. Click "Reset contraseña".
  Confirm: Loading spinner appears. After a few seconds, success toast "Correo de restablecimiento enviado para [username]".
  Check the regular user's email for the reset link (Supabase sends via email config).
  In admin_logs, a 'password_reset_sent' entry should appear.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| regular user browser→profiles table | RLS should limit SELECT to own row only |
| regular user browser→admin functions | Functions must reject with 'not_admin' |
| regular user browser→admin_logs | RLS must return 0 rows (admin-only SELECT policy) |
| admin browser→admin_logs | Admin should see all log entries (admin SELECT policy active) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-22 | Information Disclosure | Regular user queries admin_logs and receives data | mitigate | admin_logs_select_admin policy using (select private.is_admin()) — verified in RLS audit Part 2 |
| T-04-23 | Information Disclosure | Regular user reads other users' predictions or bets | mitigate | predictions_select_own and bets_select_authenticated (bets intentionally public for odds) — verified in RLS audit |
| T-04-24 | Elevation of Privilege | Regular user calls admin SQL function directly | mitigate | Admin guard raises 'not_admin' as first statement — verified in smoke test Step with non-admin user |
| T-04-25 | Tampering | Mobile tap target overlap allows accidental admin actions | mitigate | min-h-[44px] on all interactive elements; mobile QA Part 1 verifies no overlap |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No new packages in this wave; CSS-only changes |
</threat_model>
