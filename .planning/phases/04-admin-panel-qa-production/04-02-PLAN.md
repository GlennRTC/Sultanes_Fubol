<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 2: Verify live DB schema — admin_logs, functions, policies, and security</name>
  <what-built>Migration 0007_admin_panel.sql has been applied to the live Supabase project by Task 1. This creates: private schema, private.is_admin() helper, admin_logs table with RLS, profiles_select_admin and token_transactions_select_admin policies, and all five admin functions.</what-built>
  <how-to-verify>
Open your Supabase Dashboard for this project and verify ALL of the following steps.

**Step 1 — Table Editor: confirm admin_logs table exists**
- Navigate to: Dashboard → Table Editor
- Confirm "admin_logs" is listed with columns: id, admin_id, action, target_user_id, details, created_at
- If missing, re-apply migration via SQL Editor.

**Step 2 — Database → Functions: confirm all five admin functions exist**
- Navigate to: Dashboard → Database → Functions
- Confirm these function names are listed: admin_block_user, admin_grant_tokens, admin_set_match_result, create_bet_pool, admin_resolve_pool
- Also confirm private.is_admin appears in the "private" schema (may be in a separate schema selector)
- If any are missing, paste the relevant SECTION from 0007_admin_panel.sql into SQL Editor and run it.

**Step 3 — Database → Auth → Policies: confirm new policies exist**
Run in SQL Editor to verify both new RLS policies were created:

    SELECT tablename, policyname, cmd
    FROM pg_policies
    WHERE policyname IN ('profiles_select_admin', 'token_transactions_select_admin', 'admin_logs_select_admin')
    ORDER BY tablename;

Expected output: 3 rows (one for each policy name).

**Step 4 — Security check: confirm non-admin cannot call admin functions**
In SQL Editor, find the UUID of a regular (non-admin) test user:

    SELECT id, username, is_admin FROM profiles WHERE is_admin = false LIMIT 1;

Copy the non-admin user's UUID, then test calling an admin function using the Supabase JS client from the browser console while logged in as that user (not as admin):

    const { error } = await supabase.rpc('admin_block_user', {
      p_target_user_id: '<any-uuid>',
      p_blocked: true
    });
    console.log(error?.message); // Should print: "not_admin"

If the error message is "not_admin", the admin guard is working correctly.

**Step 5 — Confirm admin can SELECT all profiles (admin policy active)**
While logged in as the admin user in the browser console:

    const { data, error } = await supabase.from('profiles').select('id, username, tokens, is_blocked');
    console.log('count:', data?.length, 'error:', error?.message);

Should return ALL user rows (not just the admin's own row). If count is 1 (own row only), the profiles_select_admin policy was not applied — re-run SECTION 3 of the migration SQL.

Type "approved" when all five steps pass. If any step fails, describe which step and what you observed.
  </how-to-verify>
