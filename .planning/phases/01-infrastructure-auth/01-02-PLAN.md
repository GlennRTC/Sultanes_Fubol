<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> Supabase Auth | Untrusted credentials on signUp/signIn/reset cross here |
| browser -> profiles INSERT | Registration writes a row; RLS check is the enforcement |
| protected routes -> ProtectedRoute guard | Client-side guard for UX; RLS is the real data enforcement |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-06 | Spoofing | login brute force | accept | Supabase Auth applies built-in rate limiting on the auth endpoints; no custom auth surface added |
| T-01-07 | Tampering | client-side ProtectedRoute bypass | mitigate | ProtectedRoute is UX only; actual data protection is RLS on profiles (plan 01-01 T-01-01/02). No protected data is reachable without a valid session token |
| T-01-08 | Information Disclosure | error messages leaking account existence | accept | Reset flow shows a generic "Revisa tu correo" regardless; login shows generic "Correo o contraseña incorrectos" — does not confirm which field is wrong |
| T-01-09 | Tampering | registration INSERT spoofing another user's id | mitigate | profiles INSERT policy `with check (auth.uid() = id)` rejects any id other than the authenticated user's (plan 01-01) |
| T-01-10 | Repudiation | username squatting / duplicates | mitigate | UNIQUE constraint on username + 23505 handling surfaces a clear Spanish rejection (D-06) |
| T-01-SC | Tampering | npm dependency installs | mitigate | No new dependencies — uses only the fixed-stack packages installed in plan 01-01 |
</threat_model>
