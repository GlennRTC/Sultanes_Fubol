<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> Supabase Auth | Untrusted credentials cross here on signUp / signIn |
| browser -> profiles table | Untrusted reads/writes; RLS is the only enforcement |
| build env -> .env | VITE_ vars are bundled into client JS and are public by design |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | profiles table cross-user read | mitigate | RLS SELECT policy `auth.uid() = id` in 0001_profiles.sql — no policy exposes other rows (INF-03) |
| T-01-02 | Tampering | client writing another user's profile | mitigate | RLS INSERT/UPDATE policies use `with check (auth.uid() = id)` |
| T-01-03 | Information Disclosure | service role key leaking to frontend | mitigate | Only VITE_SUPABASE_ANON_KEY used in src/lib/supabase.ts; service role key never imported client-side |
| T-01-04 | Elevation of Privilege | client setting is_admin=true on own row | accept | Phase 1 INSERT defaults is_admin=false; admin promotion is admin-only (Phase 4). UPDATE policy lets a user write own row but no Phase-1 UI exposes is_admin; tighten with column-level grant in Phase 4 RLS audit |
| T-01-05 | Tampering | tokens manipulated from frontend | mitigate | `tokens >= 0` CHECK + no Phase-1 UI writes tokens; SQL-only mutation rule (CLAUDE.md); enforced fully when token functions land in Phase 3 |
| T-01-SC | Tampering | npm dependency installs | mitigate | All deps are from the fixed stack (react, react-dom, react-router-dom, zustand, @supabase/supabase-js, date-fns, date-fns-tz) — well-known packages; no [ASSUMED]/[SUS]/[SLOP] packages introduced |
</threat_model>
