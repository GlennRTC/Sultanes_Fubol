Add indexes `matches_datetime_idx` on `(match_datetime)` and `matches_group_idx` on `(group_name)`.
    2. `create table public.predictions` with columns: `id uuid primary key default gen_random_uuid()`, `user_id uuid not null references auth.users(id) on delete cascade`, `match_id uuid not null references public.matches(id) on delete cascade`, `home_score_prediction integer not null check (home_score_prediction >= 0)`, `away_score_prediction integer not null check (away_score_prediction >= 0)`, `tokens_wagered integer not null default 20`, `tokens_awarded integer`, `points_earned integer`, `created_at timestamptz not null default now()`, and `unique (user_id, match_id)` (enforces D-12 one-prediction-per-match lock). Add indexes `predictions_user_idx` on `(user_id)` and `predictions_match_idx` on `(match_id)`.
    3. Enable RLS on both tables. Policy `matches_select_authenticated` for select to authenticated using (true). Policy `predictions_select_own` for select to authenticated using `((select auth.uid()) = user_id)`. Add NO insert/update policies on either table — all writes go through the SECURITY DEFINER functions only (CLAUDE.md: tokens never updated directly from frontend).
    4. `alter table public.profiles add column if not exists leaderboard_points integer not null default 0;`
    5. `create view public.leaderboard_view as select id, username, tokens, leaderboard_points from public.profiles order by leaderboard_points desc, tokens desc;` then `grant select on public.leaderboard_view to authenticated;`. This view exposes ONLY safe columns — it intentionally does NOT include `is_admin` or `is_blocked` (mitigates leaderboard column-leakage threat; the profiles `profiles_select_own` RLS stays unchanged at table level).
    6. `create or replace function public.place_prediction(p_match_id uuid, p_home_score integer, p_away_score integer) returns void language plpgsql security definer set search_path = public`. Body: read `auth.uid()` into `v_user_id`; select match status; if status <> 'scheduled' then `raise exception 'match_not_scheduled'` (QUI-02 backend gate); select tokens into `v_user_tokens` for `v_user_id`; if `v_user_tokens < 20` then `raise exception 'insufficient_tokens'`; then atomically `update profiles set tokens = tokens - 20 where id = v_user_id` and `insert into predictions (user_id, match_id, home_score_prediction, away_score_prediction, tokens_wagered) values (v_user_id, p_match_id, p_home_score, p_away_score, 20)`. The exception strings `match_not_scheduled` and `insufficient_tokens` MUST match exactly — the frontend error map (Plan 03) keys on them. After the function: `revoke execute on function public.place_prediction(uuid, integer, integer) from public, anon;` and `grant execute on function public.place_prediction(uuid, integer, integer) to authenticated;`.
    7. `create or replace function public.calculate_prediction_points(p_match_id uuid, p_home_score integer, p_away_score integer) returns void language plpgsql security definer set search_path = public`. Body: `update matches set status = 'finished', home_score = p_home_score, away_score = p_away_score where id = p_match_id`; then loop over `select id, user_id, home_score_prediction, away_score_prediction from predictions where match_id = p_match_id and points_earned is null`. Per D-11: exact score (both predictions equal actual) → 3 points + 30 tokens; correct outcome (same winner sign, or both draws) → 1 point + 10 tokens; else 0 + 0. For each prediction `update predictions set points_earned = v_points, tokens_awarded = v_tokens where id = v_pred.id`; if `v_tokens > 0` then `update profiles set tokens = tokens + v_tokens, leaderboard_points = leaderboard_points + v_points where id = v_pred.user_id`. NOTE: also increment `leaderboard_points` even when tokens is 0 but points is 0 is a no-op, so only the `v_tokens > 0` branch updates profiles — but exact/correct cases always have points>0 when tokens>0, so this is consistent. After the function: `revoke execute on function public.calculate_prediction_points(uuid, integer, integer) from public, anon, authenticated;` (service-role only — no authenticated grant; prevents users self-awarding tokens, EoP threat T-02-EOP).

    Do NOT add any INSERT/UPDATE RLS policy on matches or predictions. Do NOT grant `calculate_prediction_points` to authenticated.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → Supabase RPC | Authenticated user calls `place_prediction`; token/score values cross from untrusted client |
| client → Postgres table reads | Authenticated user reads `matches`, own `predictions`, `leaderboard_view` |
| admin/service-role → RPC | `calculate_prediction_points` invoked only from service-role (dashboard/Phase 4) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | place_prediction | mitigate | Function uses internal `auth.uid()`, not a client-passed user_id — caller cannot predict as another user |
| T-02-EOP | Elevation of Privilege | calculate_prediction_points | mitigate | No `grant to authenticated`; service-role only — users cannot trigger payouts |
| T-02-TAM | Tampering | predictions table | mitigate | No INSERT/UPDATE RLS policy; only SECURITY DEFINER `place_prediction` writes, enforcing 20-token cost |
| T-02-NEG | Tampering | profiles.tokens | mitigate | `check (tokens >= 0)` on profiles + explicit pre-check in place_prediction; transaction rolls back on violation |
| T-02-ID | Information Disclosure | leaderboard_view | mitigate | View exposes only username/tokens/leaderboard_points — never is_admin/is_blocked/email |
| T-02-IDP | Information Disclosure | predictions | mitigate | `predictions_select_own` RLS restricts reads to `auth.uid() = user_id` |
| T-02-SC | Tampering | package installs | accept | No new packages installed this phase (RESEARCH Package Legitimacy Audit: none added) |
</threat_model>
