SECTION 6 — place_bet function (APU-03, APU-04, TOK-02, TOK-03):
  Signature: create or replace function public.place_bet(p_pool_id uuid, p_option_id uuid, p_amount integer) returns void language plpgsql security definer set search_path = public
  Declares: v_user_id uuid := auth.uid(), v_pool_status text, v_deadline timestamptz, v_user_tokens integer
  Logic in this exact order:
  1. SELECT status, deadline INTO v_pool_status, v_deadline FROM public.bet_pools WHERE id = p_pool_id — raise exception 'pool_not_found' if NOT FOUND
  2. IF v_pool_status <> 'open' THEN raise exception 'pool_not_open' — APU-03 gate
  3. IF now() >= v_deadline THEN raise exception 'pool_deadline_passed' — deadline check even when status is 'open' (Research open question 3 resolution)
  4. IF NOT EXISTS (SELECT 1 FROM public.pool_options WHERE id = p_option_id AND pool_id = p_pool_id) THEN raise exception 'option_not_found' — spoofed option_id protection
  5. IF p_amount < 10 THEN raise exception 'below_minimum_bet' — D-02
  6. SELECT tokens INTO v_user_tokens FROM public.profiles WHERE id = v_user_id FOR UPDATE — TOCTOU lock (CR-02 pattern from 0003, Pitfall 3)
  7. IF v_user_tokens < p_amount THEN raise exception 'insufficient_tokens' — APU-04
  8. UPDATE public.profiles SET tokens = tokens - p_amount WHERE id = v_user_id
  9. INSERT INTO public.bets (user_id, pool_id, option_id, tokens_wagered) VALUES (v_user_id, p_pool_id, p_option_id, p_amount) — UNIQUE constraint catches duplicate at DB level (APU-05)
  10. INSERT INTO public.token_transactions (user_id, amount, type, reference_id) VALUES (v_user_id, -p_amount, 'pool_bet', p_pool_id) — TOK-02 audit record
  GRANT: revoke execute on function public.place_bet(uuid, uuid, integer) from public, anon; then grant execute ... to authenticated;

SECTION 7 — resolve_pool function (APU-06, TOK-02):
  Signature: create or replace function public.resolve_pool(p_pool_id uuid, p_winning_option_id uuid) returns void language plpgsql security definer set search_path = public
  Declares: v_pool_total integer, v_winning_total integer, v_bet record, v_payout integer
  Logic:
  1. SELECT coalesce(sum(tokens_wagered), 0) INTO v_pool_total FROM public.bets WHERE pool_id = p_pool_id
  2. SELECT coalesce(sum(tokens_wagered), 0) INTO v_winning_total FROM public.bets WHERE pool_id = p_pool_id AND option_id = p_winning_option_id
  3. UPDATE public.bet_pools SET status = 'resolved', winning_option_id = p_winning_option_id WHERE id = p_pool_id
  4. IF v_winning_total = 0 THEN RETURN; — Pitfall 6 guard: no payouts when zero bets on winning option; losing tokens remain in economy
  5. FOR v_bet IN SELECT id, user_id, tokens_wagered FROM public.bets WHERE pool_id = p_pool_id AND option_id = p_winning_option_id LOOP
       v_payout := floor(v_bet.tokens_wagered::numeric / v_winning_total * v_pool_total)::integer; — FLOOR per CLAUDE.md, integers only, D-12
       UPDATE public.profiles SET tokens = tokens + v_payout WHERE id = v_bet.user_id;
       UPDATE public.bets SET tokens_won = v_payout WHERE id = v_bet.id;
       INSERT INTO public.token_transactions (user_id, amount, type, reference_id) VALUES (v_bet.user_id, v_payout, 'pool_payout', p_pool_id); — TOK-02
     END LOOP;
  GRANT: revoke execute on function public.resolve_pool(uuid, uuid) from public, anon, authenticated; — service-role only, same pattern as calculate_prediction_points
  Add SQL comment: "-- Admin calls this via Supabase SQL Editor: SELECT resolve_pool('<pool_id>', '<winning_option_id>');"

  Add a comment block after the function reminding the admin of the manual invocation pattern for Phase 3.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→place_bet RPC | p_pool_id, p_option_id, p_amount arrive untrusted from the frontend |
| admin→resolve_pool | Only callable via service role; authenticated users cannot reach this function |
| Realtime→browser | bets INSERT events delivered to all authenticated subscribers; tokens_wagered exposed in payload |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Elevation of Privilege | resolve_pool | mitigate | REVOKE EXECUTE from public, anon, authenticated — service-role only; same pattern as calculate_prediction_points |
| T-03-02 | Tampering | place_bet — TOCTOU race | mitigate | SELECT tokens FOR UPDATE before balance check; serializes concurrent calls (CR-02 pattern) |
| T-03-03 | Tampering | profiles.tokens negative balance | mitigate | CHECK (tokens >= 0) on profiles (Phase 1); place_bet raises insufficient_tokens before deduction |
| T-03-04 | Tampering | duplicate bet via two browser tabs | mitigate | UNIQUE (user_id, pool_id) on bets; DB rejects second insert with error code 23505 |
| T-03-05 | Tampering | bet amount below minimum sent from client | mitigate | place_bet validates p_amount >= 10; frontend min is UX-only |
| T-03-06 | Tampering | spoofed option_id not belonging to pool | mitigate | place_bet validates option_id EXISTS in pool_options with pool_id = p_pool_id |
| T-03-07 | Information Disclosure | bets SELECT policy permissive | accept | Tokens are non-sensitive gamification values; aggregate odds are explicitly public (D-04); individual amounts not displayed in frontend UI |
| T-03-SC | Tampering | npm/pip/cargo installs | accept | No new packages in Phase 3; package legitimacy audit in RESEARCH.md confirms zero new dependencies |
</threat_model>
